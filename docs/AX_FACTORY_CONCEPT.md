# AX Factory for Codex 構想ドキュメント

## 0. このドキュメントの目的

このドキュメントは、個人事業・小規模事業のAX（AI Transformation）を進めるために、CodexのようなAIコーディングエージェント環境と連動する「ゲーム風の事業自動化エディター」を設計するための整理資料である。

現時点では、ユーザーはすでに民泊経営のAX進捗を可視化するゲーム風ダッシュボードを作成している。
したがって、このドキュメントの目的は「ダッシュボードを新しく作ること」ではない。

目的は、既存のAXダッシュボードを発展させ、Codexのセッション、AIエージェント、業務アセット、プロンプト、実行ログ、承認フローを統合した、Codexプラグインまたは外部オーバーレイ型の「AX Factory」へ進化させることである。

---

# 1. 背景

## 1.1 当初の目的

当初、ユーザーは企業にAIエージェントを導入する支援を行おうとしていた。

しかし、企業側に対して「AIエージェントをどのように活用するのか」「具体的に何が変わるのか」をうまく伝えきれず、一度断念した。

その後、まずは自分自身の民泊経営をAXすることに方針を切り替えた。

---

## 1.2 民泊AXの取り組み

ユーザーは、自分の民泊経営業務を分解・整理し、AIエージェントが担当できる領域と人間が担当すべき領域を切り分けている。

対象となる業務には、以下のようなものがある。

* 予約管理
* ゲスト対応
* 清掃管理
* 価格調整
* レビュー対応
* 月次会計
* 設備管理
* RAG用の案内情報整備
* 問い合わせ対応
* 外部スタッフへの依頼
* 経営判断のためのレポート作成

しかし、複数領域のAXプロジェクトを同時に進めた結果、どの業務がどこまで進んでいるのかを把握しづらくなった。

そこで、各業務ごとのAXプロジェクトを可視化するダッシュボードを作成した。

---

## 1.3 既存ダッシュボードの進化

最初のダッシュボードは、業務ごとの進捗管理を行うためのものだった。

その後、作業進捗がゲームの報酬画面のように表示されると楽しいのではないかと考え、HTMLダッシュボードをゲーム風UIに書き換えた。

結果として、以下のような進化が起きた。

* 各業務がキャラクターや部門として表示される
* 部門ごとの進捗が見える
* AXレベルが上昇する
* 実績トロフィーが表示される
* 作業が進むこと自体に快感が出る
* 左モニターで進捗画面を見ながら、右モニターでCodexを操作する運用になった

ここで、ユーザーは「この2つが一体化すればよい」と考えた。

# 2. 新しい構想

## 2.1 構想の中核

構想の中核は、以下である。

> 自分の事業をゲーム画面上にデジタルツイン化し、そこにAIエージェントや人間担当者を配置し、業務アセット・プロンプト・外部ツール・実行ログ・承認フローを可視化しながら、CodexのようなAIエージェントに実作業をさせる。

これは単なるToDo管理ではない。
単なるAIチャットでもない。
単なるゲーム風ダッシュボードでもない。

目指すものは、以下である。

> Codexの作業状態を、事業自動化ゲームとして操作・監督するUI。

仮称：

* AX Factory
* AX Factory for Codex
* Business Factory
* Agent Factory
* Solo Company Simulator
* Build-a-Business
* Ops Factory

現時点では、最も素直な仮称は **AX Factory for Codex** とする。

---

## 2.2 最初のユースケース

最初のユースケースは、ユーザー自身の民泊経営でよい。

対象事業：

> 城崎温泉 柳通り ゆるり AX Factory

最初から汎用SaaSを目指さない。
まずは、自分の民泊AXを実際に進めるためのツールとして作る。

その後、以下のように広げる。

1. 自分の民泊AXで実運用
2. 民泊AXテンプレート化
3. 他の個人事業に展開
4. 地域中小企業のAX導入支援へ展開
5. Codexプラグインまたは公式機能級を目指す

3. AIエージェントの作業状態を作業員キャラとして表示する

5. 実行ログ・承認待ち・エラー・完了状態を見える化する

7. 将来的にCodex Plugin / Skill / Hook / MCPと統合する

# 4. コンセプト設計

## 4.1 ゲーム内の対応関係

AX Factoryでは、実業務をゲーム内の設備・ライン・キャラクターに変換する。

| 現実の要素      | ゲーム内表現         |
| ---------- | -------------- |
| 事業全体       | 工場・会社・街        |
| 業務部門       | 施設・部屋・ライン      |
| AIエージェント   | 作業員キャラ・ロボット    |
| 人間担当者      | 管理者・承認者・現場スタッフ |
| プロンプト      | 作業指示書          |
| Skill      | 専用工具           |
| MCP        | 外部設備との接続口      |
| Hooks      | 状態センサー         |
| App Server | Codexとの通信管制室   |
| 実行ログ       | 作業履歴           |
| 承認待ち       | ゲート・検査所        |
| エラー        | 詰まり・故障・警告      |
| 完了         | 出荷・納品・トロフィー    |
| 業務テンプレート   | Blueprint      |

---

## 4.2 エージェントを人型にするか、機械にするか

結論として、両方を使う。

ただし、役割を分ける。

### 人型・ロボ型キャラ

これは「誰が作業しているか」を表す。

例：

* CodexセッションA
* ゲスト返信エージェント
* 月次会計エージェント
* 価格分析エージェント
* RAG整備エージェント
* ユーザー本人
* 清掃スタッフ

### 機械・ライン・設備

これは「何の業務自動化が動いているか」を表す。

例：

* 問い合わせ取得ライン
* 翻訳機
* RAG参照装置
* 返信案生成機
* 承認ゲート
* 送信待ちボックス
* 月次集計機
* 価格調整研究所
* 清掃依頼ライン

### 設計方針

エージェントそのものを自動化ラインにしない。

悪い設計：

```text
ゲスト返信エージェント = 人型キャラ
清掃管理エージェント = 人型キャラ
価格調整エージェント = 人型キャラ
```

これだけでは、ただのキャラ付きタスク管理になる。

良い設計：

```text
ゲスト対応ライン
入口：Gmail / Bookingメッセージ
機械1：言語判定
機械2：施設ルールRAG参照
機械3：返信案生成
ゲート：人間承認
出口：送信候補

そこにCodexエージェントが作業員として歩いてきて、
詰まっている機械を修理・接続・改善する。
```

この形の方が、自動化が進んでいる感覚が出る。

# 6. Codexとの連携方針

## 6.1 最初からCodex本体のUI改造を狙わない

最初からCodex本体の外観を自由に変更することを前提にしない。

まずは以下の順番で進める。

1. Codexの横で動く外部オーバーレイ
2. Codexのログやセッション状態を読む
3. Codex App Server / Hooks / Skills / MCP と連携する
4. `.codex-plugin` としてまとめる
5. Codex純正機能級を目指す

---

## 6.2 Hatch Petから学ぶこと

Codexには、画面上にペットキャラクターを表示する仕組みがある。
`hatch-pet` のようなSkillにより、ペット用のアセットや設定を生成し、Codex側がそれを読み込む形になっている。

ここから学ぶべきことは、以下である。

* Codex本体を直接改造しなくても、所定の形式でアセットを生成すれば表示に反映できる可能性がある
* Codexの状態に応じてキャラクターの表示やアニメーションを変える体験が成立している
* ユーザーがCodexの内部状態を感覚的に理解できる
* 「AIが動いている感」を視覚化できる

ただし、Hatch Petは基本的にはペット表示機能であり、事業自動化全体を可視化する仕組みではない。

AX Factoryでは、これを拡張し、

```text
ペット1匹
↓
Codexセッションごとの作業員
↓
複数の業務ライン
↓
自分の事業全体のAX工場
```

へ広げる。

---

## 6.3 Claude Code側の先行例を参考にする

Claude Code側には、エージェントの状態をドット絵キャラクターやオフィス画面として表示する拡張・実験が存在する。

参考にする観点は以下である。

* AIエージェントの状態をどう取得しているか
* 実行中・思考中・待機中・出力完了をどう表現しているか
* Hooksやログ監視をどう使っているか
* キャラクター表示と実際の作業セッションをどう紐づけているか
* UIとして邪魔にならず、作業状況を直感的に見せられるか

ただし、単なる「可愛いキャラが動く」だけでは差別化にならない。

AX Factoryの差分は、以下である。

> AIエージェントの状態ではなく、事業の自動化構造そのものをゲーム画面として見せる。

# 7. 自動化ゲームから参照する要素

## 7.1 注意点

ゲームシステムを参考にする際は、具体的なアート、名称、UI、文章、キャラクター、音楽、デザインをコピーしない。

参考にするのは、抽象的なゲームメカニクスである。

「丸パクリ」する対象は、表現ではなく構造である。

---

## 7.2 Factorioから参照する要素

Factorio的な要素は、AX Factoryとの相性が非常に高い。

| Factorio的要素 | AX Factoryでの対応             |
| ----------- | -------------------------- |
| 資源          | データ、問い合わせ、予約、売上、レビュー       |
| 採掘機         | データ取得、CSV読込、Gmail取得、予約情報取得 |
| ベルトコンベア     | 情報の流れ、タスクの流れ、承認の流れ         |
| 組立機         | AIエージェント、Skill、プロンプトテンプレート |
| 研究          | 新しい自動化能力の解放                |
| 電力          | APIキー、モデル、予算、実行権限          |
| 汚染・敵        | エラー、未承認、情報不足、炎上リスク         |
| Blueprint   | 再利用可能な業務テンプレート             |

特に重要なのはBlueprintである。

AX FactoryにおけるBlueprint例：

```text
民泊ゲスト返信ブループリント
清掃依頼ブループリント
月次売上集計ブループリント
価格調整ブループリント
レビュー分析ブループリント
RAG整備ブループリント
```

---

## 7.3 Satisfactoryから参照する要素

Satisfactoryから参照すべきなのは、以下である。

* 自分の工場を持っている感覚
* 工場の中に入り込んでいる感覚
* ベルトや機械が動く気持ちよさ
* 自由に配置して改造できる感覚
* 拠点が大きくなっていく感覚

AX Factoryでは、以下に変換する。

| Satisfactory的要素 | AX Factoryでの対応 |
| --------------- | -------------- |
| 巨大工場            | 自分の事業全体        |
| ベルト             | 情報・承認・成果物の流れ   |
| 製造ライン           | 業務フロー          |
| 施設拡張            | 自動化領域の拡張       |
| 1人称の没入感         | 自分の会社の中にいる感覚   |

---

## 7.4 Shapez系ゲームから参照する要素

Shapez系から参照すべきなのは、業務を抽象的な入出力に分解する感覚である。

例：

```text
問い合わせ → 翻訳 → RAG参照 → 返信案 → 承認 → 送信
予約 → カレンダー確認 → 清掃依頼 → 完了確認
レビュー → 感情分析 → 改善点抽出 → 返信案
売上CSV → 集計 → 委託料計算 → 月次レポート
```

業務をリアルな絵で描きすぎると、汎用性が落ちる。
最初は、図形・箱・ライン・ゲートのような抽象表現でよい。

---

## 7.5 経営シミュレーションから参照する要素

工場ゲームだけでは、処理の流れは表現できても、経営判断までは弱い。

そのため、経営シミュレーションから以下の要素も取り入れる。

| 経営ゲーム要素 | AX Factoryでの対応     |
| ------- | ------------------ |
| 売上      | 月次売上、予約単価          |
| コスト     | 清掃費、手数料、外注費、API費用  |
| 人材      | 自分、清掃スタッフ、AIエージェント |
| 設備      | 客室、備品、案内端末、スタックチャン |
| 顧客満足    | レビュー、問い合わせ品質       |
| 研究開発    | 新しい自動化テンプレート       |
| 事業拡張    | 複数物件、他業種展開         |
| リスク     | クレーム、破損、誤送信、過剰自動化  |

つまり、AX Factoryは以下の合成である。

```text
Factorio的な自動化ライン
×
Satisfactory的な自分の工場感
×
Shapez的な業務抽象化
×
経営シミュレーション的なKPI管理
×
Codex的なAIエージェント実行環境
```

# 8. UI設計

## 8.1 基本画面構成

初期UIは以下の構成がよい。

```text
中央：民泊会社の工場マップ
左：業務部門一覧
右：選択中のCodexセッション / エージェント状態
下：ログ・承認待ち・エラー
上：AXレベル・売上・削減時間・未処理数
```

または、

```text
左上：事業全体マップ
右上：Codexセッション状態
左下：自動化ライン
右下：入力・承認・ログ
```

---

## 8.2 Codexセッションの表示

現在のCodexでは、左側にセッションが並んでいる。
しかし、それぞれのセッションが何をしているかは、開いて確認しないと分かりにくい。

AX Factoryでは、Codexセッションをゲーム内の作業員として表示する。

| Codex上の状態 | ゲーム表示           |
| --------- | --------------- |
| セッション実行中  | 作業員が機械を操作中      |
| 入力待ち      | 作業員がこちらを見ている    |
| レビュー待ち    | 承認ゲートで停止        |
| エラー       | ラインに警告アイコン      |
| ファイル変更あり  | 改造中の機械が光る       |
| 完了        | 出荷箱・トロフィー・AX経験値 |
| 長時間停止     | 寝ている・待機中・点検中    |

---

## 8.3 作業カードの遷移

業務カードをクリックすると、以下が表示される。

```text
業務名：
ゲスト問い合わせ対応

現在の状態：
返信案生成まではAI化済み。
送信は人間承認。

関連ファイル：
- facility_rules.md
- guest_reply_examples.md
- nearby_info.md
- faq.md

担当：
- ゲスト対応エージェント
- ユーザー本人

使えるツール：
- RAG検索
- 翻訳
- 返信案生成
- ログ保存

禁止事項：
- 勝手に送信しない
- 料金や返金について断定しない
- 未確認の周辺情報を案内しない

次の改善候補：
- Booking.comメッセージの取り込み
- 返信テンプレートの整理
- 多言語FAQの更新
```

---

## 8.4 承認フロー

メール送信、価格変更、予約関連、個人情報を含む処理は、必ず承認ゲートを通す。

承認待ち画面の例：

| 種類     | 内容            | 状態   |
| ------ | ------------- | ---- |
| メール返信案 | ゲストへの返信文      | 承認待ち |
| 価格変更案  | 週末価格を18,000円へ | 承認待ち |
| 清掃依頼文  | 清掃スタッフへの依頼文   | 承認待ち |
| 月次レポート | 売上・経費まとめ      | 完了   |
| RAG更新案 | チェックイン説明の修正   | 承認待ち |

# 9. 技術構成案

## 9.1 全体構成

理想構成は以下である。

```text
Codex
├ Skills
│  ├ ax-map
│  ├ ax-blueprint
│  ├ ax-runbook
│  └ ax-report
│
├ Hooks
│  └ セッション状態・実行ログ・承認待ちを state.json / SQLite に吐く
│
├ MCP
│  └ Gmail / Drive / Calendar / local files / booking data などと接続
│
├ App Server
│  └ スレッド一覧・実行イベント・承認イベントを外部UIに渡す
│
└ AX Factory Overlay
   ├ React / Next.js
   ├ Tauri or Electron
   ├ ゲーム風UI
```

# 14. リスクと反対意見

## 14.1 Codex本体UIを自由に改造できない可能性

CodexのPlugin機構が、任意のUI差し替えまで許しているとは限らない。

そのため、最初はCodex本体のUI改造ではなく、外部オーバーレイまたは連携アプリとして作るべきである。

---

## 14.2 ゲーム化しすぎるリスク

ゲーム風UIは作業継続の動機になる。

しかし、実務ツールとしては以下が見えなければならない。

* 実行ログ
* 失敗理由
* 承認待ち
* 差分
* 次にやること
* どのAIが何をしたか
* どのファイルが変更されたか

見た目だけが楽しいと、実務では使えない。

---

## 14.3 自動実行の危険性

民泊業務では、以下の処理は危険度が高い。

* ゲストへのメール送信
* 価格変更
* キャンセル対応
* 返金・請求
* 予約条件の変更
* 個人情報を含む処理
* 外部スタッフへの正式依頼

これらは、初期段階では必ず人間承認を挟む。

---

## 14.4 汎用化しすぎるリスク

最初から「全業種対応」を目指すと失敗しやすい。

まずは民泊AXに絞る。

その後、Blueprintを切り出すことで、他業種へ展開する。

# 17. 現時点の結論

この構想は、単なる民泊ダッシュボードではない。

また、単なるAIエージェント管理画面でもない。

最終的に目指すものは、以下である。

```text
Codexの複数セッション、AIエージェント、業務ファイル、プロンプト、実行ログ、承認待ちを、
事業自動化ゲームの画面として可視化・操作するためのAX Factory。
```

最初の実装は、以下に絞る。

```text
民泊AX Factory Overlay
```

最初の成功条件は、以下である。

```text
Codexのセッションをいちいち開かなくても、
どの業務が動いていて、
どのAIが何をしていて、
どこで止まっていて、
次に何を承認すればよいかが、
ゲーム画面上で分かること。
```

この状態が作れれば、AX Factoryは単なる見た目の楽しいダッシュボードではなく、実務用のAIエージェント操作環境になる。
